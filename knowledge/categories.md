# 副业品类库

> 用于技能映射阶段。将用户技能匹配到具体的副业方向。
> 每行字段：技能 / 启动 / 时间 / **收入三元组** / last_verified / 关键规则。

## 字段说明

- **收入三元组 `P50 / 存活 / P95 / 置信`**：替代了旧的"收入范围（月）"单一区间。详细解释见下方《收入三元组模型》。
- **last_verified**：本行数据最后一次被人工核对的日期（YYYY-MM-DD）。距今 > 90 天时，`prompts/discovery.md` 要求用 WebSearch 交叉验证后再使用。`未验证` 表示尚未做过核对，评分时不得仅凭此行数据给 4/5 分。
- **关键规则**：平台级的硬门槛 / 合规要求（如开店门槛、抽成比例、封禁规则）。为空（`—`）表示无特殊门槛或尚未录入。

---

## 收入三元组模型（必读）

**为什么不用单一"收入区间"：** 副业收入是双峰分布——90% 的人在首笔收入前就放弃，10% 活下来的中头部能赚不少。用"月入 0-10000+"作为锚点，新人默认锚定高位，形成系统性误导。

**三元组分别回答：**

| 指标 | 回答的问题 | 使用场景 |
|------|-----------|---------|
| **P50 首桶金** | 从零起步的人，第一笔有意义收入约多少？ | 建立"入场费"期望 |
| **存活率** | 起步 12 个月后还在有稳定月收入的比例 | 提示"大部分人会放弃"的现实 |
| **P95 月入** | 前 5% 做得好的人月收入 | 参考上限，**不作为决策锚点** |

**置信度（置信）：**
- `H` — 多个 Tier 1/2 来源直接核验（见 `source_trust.md`）
- `M` — 部分来源 + 推理
- `L` — 品类级估算，个案差异大，需实时 WebSearch 补数据

**评分时的使用规则（见 `prompts/scorer.md` 维度 4）：**
1. 首选 P50 + 存活率 作为收入天花板维度的依据
2. P95 只在叙事中简要提及，不作为打分依据
3. 置信度 `L` 的数据禁止作为 4-5 分评分理由；必须先 WebSearch 补数据

## 使用说明

1. 根据用户的技能列表，在每个大类中查找匹配的方向
2. 用户时间 < 5h/周时，优先推荐碎片时间可做的方向
3. 用户预算为 0 时，过滤掉启动成本 > 0 的方向（或标注需要投入）
4. **收入三元组使用规则**（与 `prompts/scorer.md` 维度 4 对齐）：
   - P50 + 存活率 作为主要锚点
   - P95 仅作叙事参考，不作为评分依据
   - 置信度 `L` 的行必须 WebSearch 补数据后才能作为 4-5 分依据
5. 引用某行数据时：
   - `last_verified` 在 90 天内：可直接使用
   - `last_verified` > 90 天或为 `未验证`：**必须用 WebSearch 交叉验证后再使用**
6. **参考案例优先**：有 `参考案例` 列的方向，评分前必须先读案例文件，优先采用案例中的具体数据
7. 此品类库不是穷举，搜索阶段可以发现不在列表中的新方向

## 数据来源与局限

- 本库的 P50 / 存活率 / P95 来自：IndieHackers Open Startups、知乎/即刻/少数派长复盘、小报童销量公开数据、Upwork 官方、程序员客栈 2026 调研帖、`cases/` 案例库
- **承认的局限**：
  - 标 `L` 的行本质是"品类级先验"而非"测量值"，应 WebSearch 补数据再采信
  - 存活率数据多来自定性描述，实际浮动 ±10%
  - 中国市场数据整体比海外难获取，偏差可能更大
- **迭代**：每次 `/side-hustle` 跑完若发现新的 Tier 1/2 数据点，回填到本库并升级置信度

## 贡献方式

- 发现数据过期：通过 `.github/ISSUE_TEMPLATE/knowledge_update.yml` 上报
- 新增品类：通过 `.github/ISSUE_TEMPLATE/new_category.yml` 提交
- 直接 PR：每行新增/更新必须附真实数据来源（含 Tier 等级），并填写 `last_verified`
